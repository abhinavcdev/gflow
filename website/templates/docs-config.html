Configuration

gflow is configured via a .gflow.yml file in your repository root. Commit this file to share conventions with your team.

Full Example

# .gflow.yml — team workflow configuration
version: "0.1.0"

provider:
  name: github
  token_env: GITHUB_TOKEN        # env var for auth token
  owner: your-org
  repo: your-repo
  # host: github.example.com     # for GitHub Enterprise

branching:
  main: main
  develop: develop
  use_develop: false              # set true for git-flow style
  prefixes:
    feature: feature/
    bugfix: bugfix/
    hotfix: hotfix/
    release: release/
    support: support/

pr:
  default_base: main
  auto_assign: true               # assign PR creator
  draft: false                    # create as draft by default
  merge_method: squash            # merge, squash, or rebase
  delete_branch_on_merge: true
  reviewers: []                   # default reviewers
  team_reviewers: []
  labels: []                      # default labels

commit:
  convention: conventional        # conventional, angular, or none
  require_scope: false
  require_ticket: false
  ticket_pattern: '[A-Z]+-\d+'
  types:
    - feat
    - fix
    - docs
    - refactor
    - test
    - chore
    - ci
    - perf

hooks:
  pre_pr:
    - "go vet ./..."
    - "go test ./..."
  post_merge: []

Provider

Currently, gflow supports GitHub as the primary provider.

Fields

  • name — Provider name. Use github.
  • token_env — Environment variable that holds your token. Default: GITHUB_TOKEN.
  • owner — Repository owner or organization.
  • repo — Repository name.
  • host — Custom hostname for GitHub Enterprise. Leave empty for github.com.

Branching

Controls how branches are created and named.

Fields

  • main — Your main/production branch. Usually main or master.
  • develop — Development branch (only used if use_develop is true).
  • use_develop — If true, features branch from develop instead of main (git-flow style).
  • prefixes — Branch name prefixes for each type (feature/, bugfix/, hotfix/, release/, support/).

Pull Requests

Controls how PRs are created and merged.

Fields

  • default_base — Default target branch for PRs.
  • auto_assign — Automatically assign the PR creator.
  • draft — Create PRs as drafts by default.
  • merge_method — How to merge: merge, squash, or rebase.
  • delete_branch_on_merge — Delete the branch after merging.
  • reviewers — Default reviewers (GitHub usernames).
  • team_reviewers — Default team reviewers.
  • labels — Default labels to apply to PRs.

Commit Conventions

Enforces commit message standards.

Fields

  • convention — conventional (feat: msg), angular (feat(scope): msg), or none.
  • require_scope — Require a scope in the commit message.
  • require_ticket — Require a ticket number in the branch name.
  • ticket_pattern — Regex for ticket numbers (e.g. [A-Z]+-\d+).
  • types — Allowed commit types.

Hooks

Run shell commands at specific lifecycle points.

Fields

  • pre_pr — Commands to run before creating a PR (e.g. tests, linting).
  • post_merge — Commands to run after a merge (e.g. deploy, notify).
hooks:
  pre_pr:
    - "go vet ./..."
    - "go test -race ./..."
    - "golangci-lint run"
  post_merge:
    - "make deploy"

Custom Strategies

Define your own branch types beyond the built-in feature/bugfix/hotfix.

strategies:
  docs:
    prefix: docs/
    base: main
    labels: ["documentation"]
    merge_method: squash
  ci:
    prefix: ci/
    base: main
    labels: ["ci/cd"]
    hooks:
      pre_pr:
        - "act -j test"

Use custom strategies with: gflow start docs update-readme
